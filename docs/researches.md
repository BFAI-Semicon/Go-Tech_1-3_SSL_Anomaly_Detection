# 工程横断ゼロショット欠陥検出：Promptable Patch Retrieval（概要書）

## 1. 背景と目的

半導体製造における欠陥検出は、工程・材料・撮像条件の差（Si 半導体／化合物半導体、
光学／SEM 等）により、教師あり学習モデルの汎化が難しい。加えて、欠陥の出現頻度が
低く、ラベル付けコストが高いという課題がある。

**自己教師あり学習（Self-Supervised Learning, SSL）**は**人手によるラベル付けを
一切必要とせず**、画像そのものに内在する構造（領域間の整合性、欠損部の再構成
可能性、ビュー変換に対する不変性など）を**モデルが自動生成する予測タスク**として
学習させる枠組みである。近年、自然画像領域では SSL によって、ImageNet などの
教師あり事前学習に匹敵する、あるいはそれを上回る汎用視覚表現が得られることが
示されている。代表的な方式は、(i) 同一画像から作った 2 つのビューの埋め込みを
一致させる**自己蒸留型 SSL（DINO／DINOv2 等）**、(ii) 入力パッチの一部をマスク
して元画素を復元させる**マスク再構成型 SSL（MAE 等）**である。**欠陥が稀で
ラベル付けコストが高い半導体製造の文脈において、ラベルを要さずに大量データから
汎用表現を獲得できる SSL は、欠陥検出パイプラインの起点として極めて適しており、
本研究はその SSL 表現を案件データに対して重み更新せずそのまま用いる。**

本研究開発の目的は、**工程・材料・撮像条件が異なる場合でも汎用的に通用する
欠陥検出モデル**を構築することであり、その実現手段として、特徴抽出器に
**Vision Transformer（ViT）**を採用する。ViT は
**Self-attention によって画像全体のパッチ間関係を一度に集約する**ため、
CNN の局所受容野では捉えにくい**スケール変化や撮像条件差に伴う構造の整合性**を
保ちやすく、装置・倍率・材料が切り替わっても表現の安定性を維持することが
期待される。加えて、ViT は前述の
**自己教師あり学習（DINO／DINOv2、MAE 等）との親和性が高く**、ラベル不要で
得た汎用表現を**未学習の工程・材料領域へ転移**させやすい。以上より、
**SSL 事前学習済み ViT を重み固定で用いる**ことが、案件ごとの再学習を要さずに
**工程・材料・撮像条件をまたいだ汎化**を実現する方法論的根拠であり、
本研究の中核を成す。

## 3. アプローチ概要（重み固定・推論時適応）

### 3.1 基本方針

- **DINOv3 を主軸に、DINO／DINOv2／MAE／汎用 ViT（事前学習済み）を特徴
  抽出器として固定**する（重み更新は行わない）。
- 超高解像度画像をパッチ分割し、大量の**パッチ特徴量**を生成する。
- 判定は教師あり分類ではなく、主として以下に基づく。
  - 正常パッチ分布からの逸脱（anomaly score）
  - ROI 注釈に基づくプロトタイプ記憶（許容／不許容）
  - 自然言語を構造化した適用条件（工程・材料・装置・期限等）

### 3.2 推論フロー（概略）

1. **特徴抽出（固定 ViT）** — 高解像度画像をタイル化・パッチ化し、パッチ特徴を
   得る。
2. **一次検出（異常スコア化）** — 正常分布からの逸脱により、欠陥候補ヒートマップ
   と ROI 候補を得る。
3. **人間フィードバック** — ROI（欠陥／非欠陥、許容／不許容）と自然言語コメント
   （適用範囲・例外条件）。
4. **構造化（LLM による JSON 化）** — 運用スキーマ（判定、適用範囲、優先度、
   有効期限など）へ変換する。
5. **補正レイヤ（Promptable Patch Retrieval）** — 欠陥候補 ROI の埋め込み
   （roi_embedding）とストア内のプロトタイプ（vit_embedding）を近傍照合し、
   類似度閾値および JSON の適用条件を満たすときに異常スコアを再構成する。
6. **最終判定** — NG、許容（False Positive の抑制）、要確認などを返す。

### 3.3 異常スコアの実装レンジ（参考）

一次検出では、パッチ特徴に対する **Mahalanobis 距離**、**k 近傍距離**
（コアセットによる PatchCore 系）、**MAE のピクセル再構成誤差**などを組み合わせ、
ヒートマップおよび ROI の切り出しに用いる。装置別・チャネル別に正常分布が
乖離する場合は、特徴量メモリをドメイン（工程・材料・装置タグ）で分割し、
適用対象と突き合わせる。

## 4. NVIDIA DGX Spark が必要な理由

**超高解像度 × 工程横断 × マルチスケール**では、推論のみでも GPU スループットが
律速となる。主因はモデルサイズより**データ側のスケール**にある：
**1 枚あたり数千〜数万パッチ**、マルチスケールの繰り返し、および
**大規模特徴量ストア**の保持とアクセス。DGX Spark の
**大容量統一メモリ（128GB LPDDR5x）**と Tensor Core の性能は、PC 級 GPU における
VRAM 枯渇とホスト転送を緩和する。LLM／VLM による補助推論を加える場合も、
実験の回転に有利となる。

### 4.1 補足：ViT 規模との関係

「ViT が大きいため GPU が必要」とだけ述べるのは適切ではない。
**ViT-Small は ResNet-50 と同程度のパラメータ**であり、モデルサイズだけが
律速要因とは限らない。一方で DINO／DINOv2 や MAE の実用チェックポイントは
**ViT-Large／Giant が多く**、モデル側のワークセットも増える。対外説明では
**パッチ総量・マルチスケール・ストア規模を主な根拠**とし、Giant の採用は
補強理由として述べるのが妥当である。

## 7. 評価計画

半導体製造は正常が圧倒的多数で欠陥が極めて稀、かつ微小欠陥が多い。画像レベルの
AUROC のみでは見逃し（False Negative）と過検出（False Positive）の非対称な
コストを反映できないため、学術指標と製造現場の運用指標を併用する多指標構成とする。

- **観点**：画像・領域単位の検出性能、見逃し（False Negative）と
  過検出（False Positive）の非対称コスト、工程・材料横断での汎化、
  フィードバック 1 件あたりの改善。
- **画像単位指標**：image-level AUROC を学術比較の基準としつつ、半導体の
  実運用に即した PG2（Pre-sorted Good at 2%：不良の 2% 見逃しを許容した際に
  良品を正しく通せる割合）・F1-Max・AUPRC を併用する。
- **領域単位指標**：pixel AUROC は正常画素が支配的で不均衡に弱いため、
  領域単位の recall を測る AUPRO[^aupro] を主指標とし、pixel AUROC は参考併記とする。
  AUPRO・pixel 指標は画素単位の正解マスクが必要なため、マスクを持つ MIIC
  （異常 116 枚にマスク付与）で算出し、マスクのない実機データには適用しない。
- **合成異常による局在化評価の補強（評価専用）**：実欠陥・実マスクが少なく
  AUPRO・pixel 指標が不安定になりやすいため、anomalib 内蔵の合成異常機能で
  正常画像に擬似欠陥と正解マスクを自動生成し、欠陥サイズ別・位置別に
  AUPRO／pixel AUROC を補って局在化能力を検証する。用途は**評価のみ**に限定
  し、特徴抽出器（重み固定）の学習や閾値・スコアの較正には用いない。合成は
  実欠陥を完全には代表しないため能力検証の参考値とし、実機データはマスク無し
  の image-level 指標・運用 KPI で別評価する二段構成を維持する。
- **運用 KPI（製造現場）**：流出率（escape rate ＝不良を良品と誤判定）と
  過検出率（overkill rate ＝良品を不良と誤判定）を運用点で報告し、必要に応じ
  ppm／DPPM[^dppm] で示す。加えてオペレータ補正回数・誤適用率・レビュー待ちキューの
  滞留などを併記する。
- **閾値・モデル選定**：見逃し（FN）と過検出（FP）のコスト非対称性に基づく
  コスト感度分析で運用点とモデルを確定する。accuracy（正解率）は極端な
  不均衡下で無意味なため主指標にはしない。
- **データ別の評価**：(1) MIIC（画素マスクあり）では image-level 指標に加え
  AUPRO／pixel 指標で局在化も評価し、必要に応じ MVTec-AD／VisA でも妥当性を
  確認する。(2) 実機データ（画素マスクなし）は image-level 指標と運用 KPI
  （流出率・過検出率）で評価する二段構成とする。
- **堅牢性の示し方**：特徴抽出器を替えたときの比較は、マスクを持つ MIIC で
  AUROC・AUPRO の両方を測る。工程横断の段階的ドメインシフト（同一工程→類似→
  異材料→異装置→異撮像）の劣化曲線は、実機データが中心で画素マスクがないため
  image-level 指標（AUROC・PG2）と運用 KPI（流出率・過検出率）で評価し、
  AUPRO はマスクのある MIIC（または小規模アノテ済みサブセット）に限定する。
  ドリフト検知後に N 件の HITL プロトタイプ追加で性能がどこまで回復するかを追う。

[^aupro]: AUPRO（Area Under the Per-Region Overlap curve）：欠陥領域ごとの
    重なり（recall）を、欠陥の大小に関わらず等しく重み付けして評価する領域指標。
    正常画素が支配的で不均衡に弱い pixel AUROC を補い、微小欠陥の検出を
    正しく測れる。慣例として偽陽性率が高い領域（FPR>30% など）を除いて算出する。

[^dppm]: DPPM（Defective Parts Per Million）：100 万個あたりの不良個数で表す
    品質指標。欠陥が極めて稀な半導体では百分率より分解能が高く、流出率
    （escape）・過検出率（overkill）を ppm 単位で表現する際に用いる。

## 9. まとめ

固定 ViT 系表現（**自己教師ありで得たセマンティックな埋め込みと、任意で MAE に
よる再構成経路**）に基づき、ラベルのない状態から開始できる欠陥検出パイプラインを
核とする。そのうえで**特徴量ストア**、**適用 scope を持つ構造化フィードバック**、
**HITL によるプロトタイプの蓄積**を重ね、モデル再学習より運用側でドリフトへ
適応できる。CNN と ImageNet 教師あり特徴と比較しても、自己教師あり事前学習済み
ViT が工程・材料横断で有利になりやすいことを、公開データと製造側 KPI の両輪で
示す。**NVIDIA DGX Spark** は、膨大なパッチとストアを前提とした評価の回転と、
将来的なマルチモーダル補強のために活用する。

## 10. 補足：DINO 特徴抽出器の入手と実装方針

本節は、特徴抽出器（DINO 系 ViT）の入手・利用方法に関する検討メモである。

- **DINO モデルはローカルで自己教師あり学習を行わず、`torch.hub`
  などから学習済み重みをロードして用いる。** SSL 事前学習は配布元
  （Meta）が実施済みであり、ローカルでは学習工程を回さない。初回のみ
  ダウンロードが走り、以降はローカルキャッシュ（既定
  `~/.cache/torch/hub`）から読み込まれる。完全オフライン運用は、clone
  済みリポジトリを `source='local'` で指定するか、`state_dict` を自前
  保存する方法で対応する。
- **ローカルで行うのは、半導体画像からの特徴抽出と、正常パッチ分布／
  メモリバンクの構築である（重み更新は行わない）。** メモリバンクは
  DINO 本体の機能ではなく PatchCore 系の仕組みであり、近傍探索
  （kNN／FAISS 等）とあわせて DINO の外側で構築する。半導体画像は
  重み学習には使わないが、正常分布の構築・プロトタイプ蓄積・評価には
  用いる。
- **現状の anomalib（v2.5.0）の PatchCore は、バックボーンとして DINO
  モデルを正式サポートしていない（既定は CNN の `wide_resnet50_2`）。**
  DINOv2／DINOv3 を timm 経由で drop-in 利用する対応は PR #3627 で
  進行中（未マージ）。暫定的には `TimmFeatureExtractor` に DINO を
  `nn.Module` として渡す方法があるが、Meta 版 DINOv2 が抽出特徴へ適用
  する layer norm の有無で精度が変わるため、CNN との比較実験では前処理
  条件を揃える必要がある。

## 11. 補足：メモリバンクの追加更新（インクリメンタル運用）

本節は、正常パッチのメモリバンクを運用中に追加更新する方法に
関する検討メモである。

- **メモリバンクは重みを更新せず、正常パッチ特徴の追記だけで更新
  できる。** 特徴抽出器（ViT）の重みは固定であり、更新は「新しい
  正常パッチの特徴ベクトルを近傍探索インデックス（kNN／FAISS 等）へ
  追加する」ことに帰着する。モデルの再学習（勾配更新）は不要であり、
  「モデル再学習より運用側でドリフトへ適応する」方針の技術的根拠でも
  ある。
- **初期構築後の定期追加という運用が可能である（例：過去 1 年分で初期
  構築し、以降は月次で追加）。** 初期構築も定期追加も作業内容は同じ
  （正常パッチ特徴の登録）であり、新しいタイプの正常品を継続的に
  取り込むことで、新規正常に対する過検出（overkill）を抑制できる。
- **追加するのは検証済みの正常のみとし、汚染を防ぐ。** 欠陥品を誤って
  正常として登録すると、その特徴に近い将来の欠陥を見逃す方向に働く
  ため、HITL による検証・スキーマ検証・監査ログで正常性を担保してから
  登録する。
- **サイズと速度は coreset とドメイン分割で管理する。** 追記を続けると
  インデックスが肥大しレイテンシが悪化するため、定期的な coreset
  再選択でサイズ上限を維持し、古い・冗長・失効した（expiry）
  プロトタイプは間引く。追加は該当ドメイン（工程・材料・装置タグ）に
  対して行い、無関係な文脈を混ぜない。
- **インデックス種別に応じた再構築を考慮する。** メモリバンクの保持と
  近傍探索を担うインデックス（推論時の異常スコア化に加え、追加更新時に
  新規点として追加するか既存点へ統合するかを最近傍距離で判定する用途
  にも使う）の方式には Flat／IVF／PQ があり、Flat 系は追記が単純だが、
  IVF／PQ など量子化系は分布が大きく変わった際にクラスタ中心の再学習・
  再構築を検討する。
  追加前後で流出率（escape）・過検出率（overkill）を監視し、追加の
  効果と汚染の影響を定量的に追う。

なお、追加更新は新規正常への過検出緩和には有効だが、欠陥が既存正常に
似る場合の見逃しは自動的には改善しない。この点は閾値設計・ドメイン
単位のメモリ純化・特徴抽出器の選定で別途対応する。

## 12. 補足：anomalib 内蔵の合成異常機能（局在化評価用）

本節は、局在化（pixel）評価を補強するための anomalib の合成異常機能の
利用メモである（評価専用。特徴抽出器の学習・閾値較正には用いない）。

- **`PerlinAnomalyGenerator`（`anomalib.data.utils.generators`）**：Perlin
  ノイズ（任意で外部の異常ソース画像）で正常画像に擬似欠陥を付与し、
  `forward(img)` で**異常画像と対応する正解マスク**を返す。ブレンド率・
  回転などを制御できる。
- **`SyntheticAnomalyDataset` /
  `make_synthetic_dataset`（`anomalib.data.utils.synthetic`）**：正常
  サンプル群から合成異常画像と画素マスクを生成し、`mask_dir` に正解
  マスクを保存する。得られた画素マスクで AUPRO／pixel AUROC を算出できる。
- **`Folder` datamodule の `test_split_mode=SYNTHETIC`**：検証・テスト分割
  に合成異常を自動投入し、局在化評価を回せる。
- **半導体向けの注意**：既定の Perlin／DTD 由来の擬似欠陥は自然画像寄りで、
  半導体の欠陥形態（スクラッチ・パーティクル・ボイド・ブリッジ等）や SEM
  の質感とはズレる。外部異常ソース画像の指定やブレンド率調整で欠陥形態を
  寄せ、ドメイン単位で妥当性を確認する。合成での AUPRO は局在化能力の
  参考値であり、実データの妥当性は image-level 指標・運用 KPI で別途担保
  する。
