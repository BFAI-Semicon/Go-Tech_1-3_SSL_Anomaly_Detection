# バージョン昇格モデル・スナップショット版管理・ドメイン単位管理（§2–§4）

> 親: [設計メモ索引](./README.md)。章番号は全体で連続（`§x.y` 参照は同ディレクトリの対応ファイル。§5→[ontology.md](./ontology.md)、§6/§9→[correction-layer.md](./correction-layer.md)、§7→[file-layout-and-samples.md](./file-layout-and-samples.md)）。

## 2. 稼働系と更新系の分離（バージョン昇格モデル）

Blue-Green 型のアーティファクト昇格モデルを採用する。

```text
[稼働バージョン Vn]  ← worker が読み取り専用で欠陥検出（メモリバンク + 構造化JSON をピン留め）
        │
        │ HITL（ROI注釈＋NL→LLM構造化→検証→監査ログ）で蓄積
        ▼
[候補バージョン Vn+1] ← 別ジョブでビルド（各ドメインの版アーティファクト再生成 / coreset再選択 / expiry間引き）
        │
        │ evaluation-framework で検証（escape/overkill 等）
        ▼
    昇格（active-manifest ポインタを原子的に切替）→ Vn+1 が新しい稼働系に
```

- **メモリバンクと構造化 JSON は 1 組（バージョンタプル）として原子的に昇格・ロールバックする。**
  補正レイヤは両方を同時に参照するため、片方だけ更新するとプロトタイプと適用条件がズレて誤補正になる。
- 稼働系は **不変バージョンを読み取り専用**で参照するのでロック競合が起きない。
- 昇格は active ポインタ（マニフェスト）の原子的な差し替えで行う。
- **稼働状態を構成する版の軸は粒度が異なる**（`§2.1` の単一プール前提の帰結）:
  - **メモリバンク＝グローバル 1 軸**。全ドメイン共有の単一プールなので per-domain の版は持たず、`memory_bank`
    スナップショット（例 `mb-2026-07-01`）で 1 本管理する（マニフェストが保持。§7.1）。
  - **構造化 JSON＝per-domain 軸**。各ドメインの版アーティファクト（`domains/<domain_id>/<domain_version>.json`。§4）。
  - **優先順位明示上書き＝任意のグローバル 1 軸**。版付き不変な `priority.artifact` をマニフェストから
    最大 1 つ参照する。参照が無い場合は明示上書きなしとする（§9.1）。
  - よって**バージョンタプル ＝ `(グローバル memory_bank snapshot, priority artifact または未指定,
    各ドメインの版アーティファクト)`**。このタプルをマニフェスト 1 本に束ね、マニフェスト自体も版付き不変で
    アーカイブする（§7.1）。過去状態の完全再現は、`active-manifest` ポインタを当該マニフェストへ向けて行う。
- **サイズ制約と切替方式**：全ドメインで同時稼働できるメモリバンクは 1 つだけ（サイズが大きい）。よって
  プロセス内 Blue-Green（新旧 2 版の同時メモリ保持）は行わず、**メモリはシングルライブ、ディスクは版付き
  不変アーティファクト**とする。昇格＝候補をビルド・検証後に**バッチ時リロードで差し替え**、ロールバック＝
  旧アーティファクトの再ロード（反映はバッチのため短時間ダウンタイムは許容）。
- **メモリバンクの版は単一親の系譜（tree）を持つ**：各 `memory_bank` スナップショットは自身をビルドした際の
  親スナップショット `parent_bank_snapshot_id` を不変メタに刻む（root は `null`）。**merge は許可しない**ため
  親は高々 1 つ。ロールバック後の再ビルドで分岐（兄弟）が生じるので、版の新旧は連番の大小ではなく系譜で判断する
  （§4 のロールバック整合、bank 互換の祖先判定）。

### 2.1 メモリバンク側のプロトタイプモデル（タプルの片側）

- **2 層構成**：特徴ベクトルと `prototype_id` は FAISS（`faiss-cpu`）が保持し kNN を担う。ドメインタグ等の
  メタデータは FAISS 外の**メタデータ層**が `prototype_id` をキーに保持する（`patch-feature-store` の
  索引層／メタデータ層分離）。`prototype_id` は FAISS の int64 ID そのものであり、補正レコードの
  `match.prototype_ids` でも JSON 整数（int64）として持つ（文字列化しない。§8.1）。
  - 間引きから保護したいプロトタイプは、**メタデータ層の `pinned` フラグ**で表現する（補正レコードに依存
    させない）。coreset は自レイヤ（`pinned` 等）のみを見て自己完結し、依存は「補正→メモリバンク」の
    一方向を保つ。
- **`prototype_id` は安定・不再利用**。coreset 再選択や expiry で除外しても ID は再利用せず、論理的に
  retire（tombstone）する。これにより補正レコードの `match.prototype_ids` が誤解決せず、§9 規則2 の
  参照整合（remap／非適用）と整合する。なお**未参照のプロトタイプは孤児ではなく正常集合の通常メンバー**
  であり、参照がないことを理由に除外しない（大多数のプロトタイプはどの補正レコードからも参照されない）。
- **`domain_id` は集合（1..N）**。プロトタイプは特定ドメインのパッチ由来だが、分布が重なるドメイン間で
  近接点が集約（マージ）されると、集約点は**由来ドメインの和集合**を持つ。
  - マージは最近傍距離のしきい値で判定する（`patch-feature-store/brief.md`）。ただし**無関係な文脈は
    混ぜない**（`researches.md §11`）。多重 `domain_id` は「単一ストアで実際に分布が重なるドメイン間」で生じる。
  - 任意でドメイン別の寄与カウント/重みを持たせ、較正・coreset に使う。
- **一次判定はドメイン非依存（汎用）**。本開発の目標は工程・材料・装置によらない汎用欠陥検出であり、
  一次判定は **メモリバンク全体（プール）を正常集合**として kNN する。`domain_id` によるハード
  フィルタは行わない。したがって**メモリバンクに存在しない未知ドメインの画像**でも、既存プロトタイプ
  との近傍距離から異常スコアを算出でき、判定を出せる（汎用なのは凍結 SSL 特徴抽出器と単一プールの
  正常集合）。
- **`domain_id` は取得のフィルタではなく、補正レイヤと較正のためのメタデータ**。用途は
  (1) 補正レコードの適用スコープ照合（§6 `match.scope`）、(2) ドメイン別のしきい値較正・寄与重み、
  (3) 分布が大きく乖離する場合の任意の正常分布のドメイン別推定（参照集合のドメイン別分割。`researches.md §3.3・§8`）に限る。一次判定の
  「どのプロトタイプを引くか」を絞る目的では使わない。
- **マージ時の id 対応**：集約で新 `prototype_id` を発番する場合は旧→新の対応表を残し（実質 remap）、
  既存補正レコードの `match.prototype_ids` の参照追跡を保つ。

#### 関連研究による裏付け（ドメイン別 vs 全体で判定）

近年の欠陥検出研究は「クラス（ドメイン）ごとに別モデル/別メモリバンク」から
「1 モデルで全クラスを判定する統合（unified / multi-class, MUAD）」へ移行しており、
本設計の「ドメイン非依存・単一プールで一次判定」は、この潮流および未知ドメイン対応の要件と整合する。

- **古典 PatchCore はカテゴリ別**。カテゴリごとに別メモリバンクを持つ one-model-per-class が標準運用
  （Roth et al., CVPR 2022, `amazon-science/patchcore-inspection`）。
- **UniAD（NeurIPS 2022）** が全クラス 1 モデルの MUAD 設定を提唱。当初は再構成系の identity mapping
  問題でクラス別に劣後。
- **Dinomaly（CVPR 2025）／Dinomaly2（arXiv 2510.17611, 2025）** で統合モデルがクラス別 SOTA に
  到達（MVTec-AD 99.6%）。Dinomaly2 は single/multi-class・few-shot・2D/3D/multi-view を 1 枠組みで
  扱う full-spectrum を主張し、「専用モデル乱立＝デプロイ障壁」を明確に問題視。
- **未知ドメインへの汎化**：WinCLIP / APRIL-GAN / AnomalyCLIP 等のゼロショット系や Dinomaly2 の few-shot は、
  そのクラス用メモリバンクを作らずに未見カテゴリへ判定を出す。＝「メモリバンク未収録の未知ドメインでも判定」要件と同方向。
- **単一プールの注意点**：クラス間干渉で微小欠陥スコアが薄まりうる。対策として強力な foundation 特徴抽出器、
  位置・近傍条件付け（PNI, ICCV 2023）、多メモリ構成（TMUAD, arXiv 2508.21795）などがある。本設計では
  `domain_id` を（フィルタではなく）ドメイン別しきい値較正・寄与重み・任意の正常分布のドメイン別推定に用いてこの弱点を補える。

## 3. バージョン管理方式：スナップショット＝真実（版付き不変アーティファクト）

- **真実（source of truth）は版付き不変のスナップショット**（各ドメインの版アーティファクト・メモリバンク
  スナップショット・優先順位アーティファクト・マニフェスト）であり、これらをアーカイブしてピン留めする。
- **バージョン軸 = 各アーティファクトの識別子とマニフェスト参照**。ドメインの版はドメイン内で単調増加する整数
  `domain_version`（§4）で識別し、稼働状態は `active-manifest` が指すマニフェストのタプル全体で一意に決まる。
- **ドメインの作業ファイルは削除・変更可能なビルド時点の必要最小構成**とする。各ビルドで単一
  ライタがドメインの JSON を全生成し、検証後に**版付き不変アーティファクトとして publish**する（§7）。
- **再現性・監査・ロールバックはアーカイブ済みスナップショットが担う**（可変な作業ファイルではない）。過去の
  有効ルール集合はその版アーティファクトが保持し、詳細な来歴（誰が・いつ・元の表現）は独立監査ログが持つ
  （`source_ref`。§6.3・§11）。
- `ontology_version`: 自社の複合語彙版（`semicont` は git ref ピン＋`proj` は自社版。§5.1）。要素単位で
  authored 版を自己記述し、ドメインファイルの `domain_source_ontology_version` は domain CURIE の解釈元を示す（§5.1）。
- **自前の JSON 構造版（`*_schema_version`）は持たない。** 我々が変更タイミングを支配でき、
  変更頻度も低いため。ただし将来の破壊的変更に備え、次を規約とする:
  - **「`schema_version` フィールドが無い＝バージョン `1.0`」と解釈する。**
  - 破壊的なスキーマ変更を行う時点で初めて `schema_version` を導入する。
  - スナップショット・`priority.json`・マニフェストはいずれも版付き不変アーティファクトとして公開し、既存
    ファイルを書き換えず新しい版として publish する。

### 3.1 削除の表現：レコードの実削除

- **削除は当該要素をドメインファイルから実際に取り除く**ことで表現する（作業ファイルが可変・最小構成のため。§3）。
  削除フラグ（tombstone）は用いない。
- 削除された要素は有効集合に現れないので、重なる広域ルールがあればそれにフォールバックする（真の削除と一致）。
- 意味は「今後の廃止（deprecation）」であり「過去の撤回」ではない。誤登録などの来歴は独立監査ログに記録する
  （レコード本体には持たない）。
- 再登録は、同じ論理要素に新しい `element_id` を採番して要素を書き戻せば可能。
- **削除後も `element_id` は再利用しない**（§6.3）。削除済み要素の記録を残さないため、非再利用は専用の単調
  カウンタ（バージョンタプルの外に置き、ロールバックで後退しない state）と独立監査ログの backstop で保証する。
  次の `element_id` を現行ドメインファイルの最大値やアクティブなマニフェストから導出してはならない（削除・
  ロールバックで後退し衝突するため）。
- **`KeepPrimary`（一次判定を保つ能動マスク）は削除ではない**。削除された要素は有効集合から消えて広域ルールに
  フォールバックするが、`KeepPrimary` は有効集合に残り specificity（§9.1）で広域ルールを遮蔽する（§6.1）。

## 4. ドメイン単位のバージョン管理

- **1 ドメイン = 1 JSON（版付き）。** ドメインの版は**ドメイン内で単調増加する整数 `domain_version`** で識別し、
  版ごとに不変アーティファクト `domains/<domain_id>/<domain_version>.json` として publish する（§7）。
- **構造化 JSON の昇格・ロールバックはドメイン単位で扱える**（他ドメインに波及させない）。ただし稼働状態は
  マニフェスト（タプル全体）で一意に決まるため、ロールバックは目的の異なる 2 操作を区別する（§7.1）:
  - **タプル全体のロールバック**：`active-manifest` を過去マニフェストへ差し替えるだけ（新マニフェストは作らない）。
    ドメイン単位ではなく、bank・priority・全ドメインが当時のタプルへ**一括で**戻る。
  - **部分ロールバック（ドメイン単位）**：現在の他軸（bank・priority・他ドメイン）を維持したまま 1 ドメインだけを
    旧 `domain_version` へ戻す。過去マニフェストには当時の他軸が丸ごと入っており流用できないため、旧ドメイン版と
    現在の他軸を組み合わせた**新しいマニフェストを発行**（`manifest_version` は最大＋1）してポインタを向け直す。
- 稼働系は起動時にマニフェストが束ねる**全ドメインのアーティファクトをロード**し、`domain_id` キー＋specificity
  チェーン（`any`・上位工程クラスを含む §4.2 の合成順）を張った**合成インメモリ索引**を構築する。クロスドメイン
  競合（§9.1）と稼働状態あたり全体1本の `priority`（§9.1）を正しく解くため、広域ドメインも常に索引へ含める。
  将来メモリバンクを物理分割する場合（§13）は「担当具体ドメイン＋適用可能な広域ドメイン」の部分ロードへ切り替える。
- **メモリバンクは per-domain 版を持たない（グローバル 1 軸。§2・§2.1）。** ここに粒度の非対称がある:
  - **per-domain 操作（部分ロールバック含む）**：あるドメインの構造化 JSON だけを別 `domain_version` に昇格／
    ロールバックし、メモリバンク（グローバル snapshot）と他ドメインは据え置く。現在の他軸を組み込んだ新しい
    マニフェストを発行して実現する（過去マニフェストへの差し替えでは他軸も戻ってしまう。§7.1）。
  - **グローバル操作**：メモリバンク snapshot の差し替え／巻き戻しは**全ドメインに波及する**（全ドメインが同じ
    プールを参照するため）。タプル全体のロールバック（過去マニフェストへのポインタ差し替え）もこれに該当する。
  - **ロールバック整合**：あるドメインの JSON だけ巻き戻しても、`prototype_id` 安定＋非再利用＋§9 規則2
    （dangling は補正側で remap／非適用）により誤解決しない（該当補正が非適用になるだけ）。ただし「当時の判定を
    完全再現」するには、その時点の **memory_bank snapshot も併せてピン留め**する必要がある（§2 のタプル）。
- **ドメイン版とメモリバンク版の互換（bank 祖先判定）**：各ドメイン版は、ビルド時に使ったメモリバンクを
  `built_against_bank_snapshot_id` として刻む（`match.prototype_ids` の参照先が存在した版。§6.3）。稼働タプル
  構築時は、**active な `memory_bank` から `parent_bank_snapshot_id` を root（`null`）まで遡り、
  `built_against_bank_snapshot_id` に到達できること**を検証する（到達不可＝別枝または古すぎる bank で、参照が
  解決しないため拒否する）。到達しても coreset／expiry で個別プロトタイプが除外されている場合は §9 規則2 で
  remap／skip する（祖先判定はゲートであって包含保証ではない）。publish 時のプロトタイプ実所属照合や到達不能枝の
  GC は当面行わず、必要時に追加する（§13）。

### 4.1 ドメインキーと 4 桁スラッグ

- ドメインキー = `(process, material, equipment, unit_of_work)` の複合キー。
- **正キーはタプルから初回生成する `domain_id`（ハッシュ）で、ディレクトリのキーにも `domain_id` を使う**
  （`domains/<domain_id>/`）。CURIE 改名で表示ラベルが変わっても版履歴が 1 ディレクトリに集約され続ける。
- スラッグ（`process__material__equipment__unit`、固定順序 4 桁）は**人間可読ラベル**として `manifest` に保持する
  （パスの権威キーにはしない）。
  - `site`（Facilities）はキーに含めない（4 桁に確定）。
  - `Measurements` はキーではなく `match.scope` の条件に使う。
  - 指定しない軸は **明示トークン `any`** で埋める（空欄を `_` で埋める方式は非推奨：判読・パースが脆い）。
- **`domain_id` は初回採番後は不変の不透明キーとして凍結する。** CURIE タプルは記述的属性に過ぎず、
  オントロジー改名等で CURIE が変わっても `domain_id` は再計算しない（既存アーティファクトとの連続性を保つ。§5.1）。
- ドメインファイルは domain CURIE タプルを記述した作成時の `domain_source_ontology_version` を保持する。
  スナップショットには新旧の domain 表現を `domain_representations_by_ontology_version` へ版キーで格納し、
  正規化先を `target_ontology_version` で示す。specificity 照合とドメイン合成には目標版に対応する
  domain 表現だけを使い、要素の authored 版は要素単位の `ontology_version` で確認する（§5.1、§8.3）。

### 4.2 「工程全体」の表現（2 通り）

1. **特定工程を材料・装置問わず適用** → `process` は固定、他軸を `any`。
   例: スラッグ `drie__any__any__wafer`（CURIE 側は `material: "*"`, `equipment: "*"`）
2. **工程ファミリ全体に適用** → ワイルドカードではなく **上位クラス CURIE** を使う。
   例: スラッグ `dryetch__any__any__wafer`（`process: "semicont:DryEtchProcess"`）

稼働系は「装置指定 → 材料指定 → 工程全体（`any`）→ 上位工程クラス」の順（specificity 優先）に
合成して適用する。これは §9.1 の優先順位チェーンの第 1 段（specificity）と同一の考え方。
